Section 4: Error Handling & Edge Cases

What can go wrong per tool, and how fy recovers gracefully

Per-tool failure modes

Tool
Failure
Recovery
🌐 Web
Grounding returns no results or Gemini quota exceeded
Fall back to plain Gemini answer (no sources). Show "Sources unavailable" label.
▶ YouTube
Video has no captions / transcript disabled / private video / API quota
Return user-facing message: "Cette vidéo n'a pas de sous-titres disponibles." For API quota, show retry prompt.
🖼 Image
Imagen rejects prompt (safety filter) or Vertex quota exceeded
Surface safety message in fy's voice: "Je ne peux pas générer cette image." Never expose raw API error.
🎤 Voice
Microphone permission denied / empty audio / transcription fails
Show inline prompt: "Accès au micro refusé." On empty audio, silently reset. On transcription failure, show toast.
📎 File
File too large (Gemini inline limit ~20 MB) / unsupported type
Validate client-side before upload. Reject with "Fichier trop volumineux (max 20 MB)" or "Format non supporté (PDF, JPEG, PNG)."

Global patterns

API errors never reach the UI raw
All route handlers wrap responses in try/catch. Error messages are always translated into FR/MG/EN by locale. Status codes are normalized to 200 with an error field so the frontend has one code path.
No double-loading states
The existing isTyping flag covers all tool requests. The tool badge ("Web Search · Searching…") replaces the generic typing dots while a tool is active — one consistent loading pattern.
Intent misfire escape hatch
If detectIntent() fires the wrong tool, the response will be obviously wrong. The tool pill stays togglable — user can manually force the correct tool and resend.
YouTube transcript fallback
If captions aren't available, fy analyses the video title + description + metadata only — still returns useful context rather than a hard failure.
