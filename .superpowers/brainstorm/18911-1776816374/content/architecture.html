fy Tool Architecture — Design

Section 1 of 4: System overview & data flow

Request flow

User message / audio / file → detectIntent() → Show tool badge
🔵 /api/chat
Plain conversation — no tool intent detected. Passes full history to Gemini 2.5 Flash, returns text.
🟡 /api/tools/web
Gemini with Grounding enabled. Returns answer text + array of sources [{title, url, snippet}].
🟢 /api/tools/youtube
YouTube Data API (search or extract ID from URL) → fetch transcript → Gemini summarises. Returns summary + video metadata.
🟣 /api/tools/image
Imagen 3 via Vertex AI. Translates user prompt → generates image → returns base64. Rendered inline in chat.
🔴 /api/tools/transcribe
Receives audio blob → sends to Gemini as multimodal input → returns transcribed text. Result is injected back into the chat input.
🔵 /api/chat (multimodal)
Same /api/chat route, but message payload includes base64 file + mimeType. Gemini reads PDFs, JPEGs, PNGs natively.

Intent detection — client-side, zero latency

A detectIntent(message) utility runs on the message text before sending. It returns one of: "web" | "youtube" | "image" | "chat". Voice and file are triggered by UI events (mic button, paperclip), not text patterns.

→ web
"recherche…", "search for…", "what is…", "latest news", "actualités", "vaovao"
→ youtube
youtube.com/watch, youtu.be/, "resume cette video", "summarise this video", "résume"
→ image
"generate image", "génère une image", "crée une image", "draw", "dessine", "sary"
