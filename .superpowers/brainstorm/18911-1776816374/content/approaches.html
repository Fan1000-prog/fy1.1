3 Approaches for fy's Tool Architecture

How the backend orchestrates web search, YouTube, image gen, voice, and file analysis

A

Vercel AI SDK + Gemini Recommended

Migrate the API route to use @ai-sdk/google + streamText with typed tool definitions. Each tool (webSearch, youtubeAnalysis, generateImage, analyzeFile) is declared once — Gemini decides when to call them. The SDK handles streaming, tool invocation state, and multimodal payloads automatically.

Browser → stream chat → /api/chat (streamText) → toolCall → execute(tool) → result → Gemini synthesizes → stream to UI

Pros

  • Streaming first-class via useChat
  • Tool invocation state built-in (show badge automatically)
  • Multimodal files & audio handled natively
  • Easiest to add new tools later

Cons

  • Adds @ai-sdk/google dependency
  • Requires refactoring existing /api/chat route
B

Raw Gemini Function Calling

Keep the existing raw fetch to Vertex AI, but add Gemini's native tools / functionDeclarations to the request body. The API route runs an orchestration loop: call Gemini → handle functionCall → execute → feed result back → repeat until final text response.

Browser → POST → /api/chat → loop until text → Vertex AI raw fetch → JSON response

Pros

  • Zero new dependencies
  • Stays 100% Google ecosystem

Cons

  • No streaming — full response before UI updates
  • Orchestration loop is tricky to get right
  • Multimodal / file handling is manual
C

Intent Routing (Separate API Routes)

Keep the main Gemini chat route. Add dedicated routes per tool (/api/tools/web, /api/tools/youtube, /api/tools/image). The frontend pre-classifies intent (regex + keyword matching on the message) and calls the right route, showing the tool badge before the request completes.

Browser classifies intent → /api/tools/[tool] or /api/chat

Pros

  • Very simple to reason about
  • Easy to debug each tool in isolation

Cons

  • Client-side intent detection is fragile
  • Multi-tool requests (search + image) don't compose
  • No streaming on main chat
